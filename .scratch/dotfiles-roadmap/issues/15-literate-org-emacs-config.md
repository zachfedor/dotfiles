# Convert Emacs config to literate org via Doom tooling

Status: needs-triage

## What to build

Convert the Doom Emacs private config (`~/.dotfiles/doom/` — `config.el`,
`packages.el`, `init.el`) into a **literate org configuration** using Doom's
built-in support: a `config.org` whose source blocks tangle to the real `.el`
files (Doom autotangles `~/.config/doom/config.org` on `doom sync`). Goal:
prose-documented, narratively-organized config — the *why* lives next to the
*what*.

## Open questions (triage / grill)

- **Which files go literate** — just `config.el` → `config.org` (the common Doom
  pattern; `init.el`/`packages.el` often stay plain because they're read very
  early)? Or all three? Confirm Doom's autotangle covers what you target.
- **Tangle vs source-of-truth** — `config.org` becomes canonical; tangled
  `config.el` is a build artifact. Decide: commit the tangled `.el` too (works
  even if tangle hasn't run / on a fresh box before first sync), or gitignore it
  and rely on `doom sync` to regenerate. Affects the nix passthrough below.
- **Nix passthrough interaction** — `doom/` is placed out-of-store via
  `mkOutOfStoreSymlink` (ADR-0002, Doom hand-managed). A literate setup adds a
  tangle step; make sure first-boot on a new host still produces working `.el`
  (ties to the new-host runbook in `docs/new-host.md`).
- **Heavy existing config** — `config.el`/`packages.el` already carry the #07 org
  workflow (capture/refile/agenda/archive fns). Migration must preserve all of it
  byte-for-byte behavior; do it as a mechanical move into blocks first, *then*
  add prose — don't rewrite logic during the conversion.
- **Cross-host** — both hestia + athena run the same Doom config; verify the
  literate setup tangles cleanly on both (athena's Doom is hand-bootstrapped).
